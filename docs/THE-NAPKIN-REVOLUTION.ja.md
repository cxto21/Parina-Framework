# ナプキン革命
## 持続可能でレジリエントなソフトウェア開発のための提案

ネルソン・ロハス・ヌニェス 著

まずはじめに、お詫びいたします。気分を害するつもりはありませんし、誰かを責めるつもりもありません。偉大な賢者ベロニカが言ったように、あなたのツールはそれで十分に機能しているからです。

長い間、私はキーボードに向かってコードを書いていました。社会に出て働き始めるとすぐに、ベストプラクティスを取り入れ、その時使っていた言語の典型的な10の過ちを犯さないようにと心がけ、しばらくの間そのように過ごしていました。

ある時、Web開発に手を出してみたくなったのですが、キーボードを叩き続け、同じ理論を使い、オフィスでやっていたことと同じことを繰り返す日々に、すっかり退屈してしまいました。何か新しいもの、理解しやすいものを求めていたのです。

そんな中、「求めよ、さらば与えられん」の通り、多くの奇跡的なツールに出会いました。開発プロセスを楽しくし、ある種の調和やひらめきをもたらしてくれるコードジェネレーターたちです。

そこで私は大学に戻り、新しい世代に向けてこれらの奇跡的なツールが存在することを伝えることにしました。何度も招待され、自分の街の外へも足を伸ばしました。

その後、スペイン語のコミュニティに参加し、いくつか動画を撮影してYouTubeにアップロードしました。趣味や好奇心、そして自分自身の問題解決のためにプログラミングを始めた初心者たちに、いろいろなことを説明するために時間を使いました。

このコミュニティのためにいくつかの記事を書きましたが、私には教える者としての憧れがまだありました。新しい世代に、車輪がどのように作られ、どのように設計され、その中に何があるのかを伝えたかったのです。

意見が一致しなかったため、私は個人で活動することにしました。たくさんの愛をくれたコミュニティにあったものと似たような車輪をいくつか作り、複製を試みましたが、それほど良いものはできませんでした。何と言っても、彼らは私より何年も先を行っていたのですから。

途中で立ち止まり、ある日モダンなPHPとSOLID原則に関するコースを購入しました。それからアーキテクチャ、CQRSパターン、そしてメンテナンス性の高いコードに関する本を何冊か読みました。

そして私は、誰の頭にもすっと入るような、シンプルなアイデアに立ち返ることにしました。

何度も処理を重ねるうちに、頭の中に電球が灯りました。「私たちがやっていることはすべて、テキスト処理にすぎない」という結論に達したのです。斬新でしょう？ Unixの経験がある人なら、それは誰もが理解できることです。しかし、街を歩いている一般の人々、その時々の輝かしい奇跡的なツールの広告を受け取っている人々は……それを理解しているでしょうか？

こうして、紙ナプキンのサイズに収まる図のフレームワークを構築するというアイデアが生まれました。

しかし、おもちゃではありません。Secure Code Warrior のコースが私の心に深く刻まれていたからです。安全で、高品質で、テスト可能であり、何よりも理解しやすく拡張可能なものである必要がありました。

小さな図にしては、制約が多すぎますよね？

最初のステップは、リクエストの集中器であるフロントコントローラー (front controller) です。名前は立派ですが、サーバーに対して、届くすべてのアクセスを同じファイル（シンプルな index.php）に通すように指示する仕組みにすぎません。

これで、すべてのトラフィックが集中器に集まります。

第2のステップは、各リクエストが単一の責任を持つこと（SOLIDのS）であり、ここでは極限までシンプルにしました。ハンドラー (handler) です。それだけです。

少し立ち止まりましょう。ハンドラー (handler) はリクエスト (request) で送られてきたものを受け取り、処理してテキストで応答します。つまり、レスポンス (response) を生成するのです。

そして理論上は……これですべてです！

「でもネルソン！セキュリティがあり、拡張性があり、コードの品質が高く、テスト可能だって言ったじゃないか！」

SOLIDの原則の一つにインターフェース (Interfaces) の使用があります。これは一見役に立たない設計図のようなものですが、コントラクト（メソッド）を持っており、それを使用する者はそれを満たす（実装する）必要があります。

では、なぜインターフェースが必要なのでしょうか？ 簡単に言えば、私の見解では、同じような振る舞いを持ちながらも、その振る舞いの処理にいくつかのバリエーションがあるもの（多態性）を作成するための基盤だからです。

そしてインターフェースは何を可能にするのでしょうか？ 拡張性と標準化を強制することです！

さて、いくつか基本的なパーツがあります。ユーザーがアクセスできるルートを定義するもの（ルーター Router）が必要です。ルートのリストに従って届いたリクエストを振り分けたり、エラーメッセージをトリガーしたりするもの（ディスパッチャー）が必要で、私の場合はこれをカーネル (Kernel) と呼ぶことにしました。

.NET、Java、またはJSPで作業したことがあるなら、Request（リクエスト）とResponse（レスポンス）と呼ばれるものが存在することを想定するでしょう。これらはそれぞれ「クライアントからの要求」と「こちらから返す応答」を意味します。

リクエスト of 例：
- シンプルなリンク
- パラメータ付きのリンク
- テキスト付きのフォーム
- 添付ファイル付きのフォーム

レスポンス of 例：
- ステータスコード
- プレーンテキスト
- HTML
- JSON
- XML
- 添付ファイル
- リダイレクト
- などなど。

では、セキュリティは？
ある日、リクエストが入ってくる際に、処理を進めるかどうかを決定する一連の条件が必要であることに気づきました。それは宮殿の衛兵であり、開発用語で言えばミドルウェア (middleware)（サンドイッチのハンバーグのようなもの）です。パンの反対側にたどり着くには、必ずそれを通らなければなりません。

ミドルウェアは何をするのでしょうか？ リクエストをチェックし、バリデーションを行い、内容を修正またはクリーンアップし、攻撃ベクトルを排除するのを助けます。

しかし、リクエストの途中にそれを阻止しようとする衛兵のリストが存在することもあれば、まったく無いこともあります。それはリクエストの複雑さや、応答を期待する目的地によって異なります。

結果として、ルートを定義するたびに私はこう言います。「いいですか、もし誰かがA門宛てのリクエストを黄色い道 (GET) を通って持ってきたとして、その門に衛兵が必要ないなら、私は何も邪魔をせず、門までの地図（リクエストを受け取って処理するハンドラー handler）を渡します」

逆に、もし誰かがB門宛てのリクエストを持ってきて、その門には『Wonka』というフレーズが書かれた金の招待状が必要な場合、私はその招待状を尋ねてチェックする門番（ミドルウェア middleware）を配置します。門番が通過を許可した場合、B門までの地図（処理を行って結果を返すハンドラー handler）を渡します。

各ハンドラー (handler) は必要な詳細情報とともにリクエストを受け取り、適切に処理を行いますが、常に、常に、常に、別のインターフェースから派生したレスポンスタイプである Response を送信します。

これでレスポンスが拡張可能になりました。契約（Response インターフェースのメソッド）を満たしている限り、必要な種類のレスポンスを作成できます。

無駄なCPU消費を避けるため、門番が条件を満たしていないと判断した場合、一瞬で排除され、振り返ることもありません。リクエストとその履歴は時間の彼方に消え去ります。

それでは、ナプキン上の説明の締めくくりとして、順を追った手順を示します：

1. リクエスト (Request)
2. フロントコントローラー (Front Controller)
3. ルートの読み込み (メソッド, ルート, 担当ハンドラー, ミドルウェアのリスト) (Route)
4. ルートリストを使用してリクエストと一致させるディスパッチャー (Kernel)。ミドルウェアが存在する場合は、それらを順次実行します。全員が許可を与えれば処理を続行し、一人が同意しない場合は即座に回路を遮断します。
5. 出力 (Response)

これが Parina Framework です。線形で、厳格で、拡張可能で、高度にテスト可能であり、後からメンテナンスを行う人に対して極めて親切なシステムです。なぜなら、各リクエストは単一の handle メソッドを持つハンドラーによってのみ応答されるからです。

だからこそ、ナプキンに収まるのです。精神的に非常に消化しやすく、何かを改善する必要があると感じるたびに、頭の中は常に同じ結論に達するはずです。「もう一つミドルウェアを追加しよう」。なぜなら、彼らがフローを担当し、危険な入力を阻止し、1秒あたりの過剰なリクエストを制限し、権限のないユーザーをリダイレクトする役割を担っているからです。

そして、あの小さな紙切れは私に予期せぬ教訓を与えてくれました。極めて高速なリクエスト/レスポンスサイクルと、信じられないほど小さなメモリフットプリントです。

512MBのRAMとセレロン (Celeron) プロセッサを搭載した2003年のマシンで、Parinaを実行しているサーバーは、わずか約40KBのメモリを消費しながら、1秒間に200回以上のリクエストに応答することができました。

こうして私は、概念的に明確で、安全で、テスト可能で、理解しやすいツールを手に入れたと結論づけました。そして、必要なリソースがごくわずかであるおかげで、業界から見捨てられた人々や、インターネット接続が遅いコミュニティ、あるいは15年以上前のコンピュータを使用している人々のためのツールになり得るのです。
