# 餐巾纸革命
## 一种实现可持续和弹性软件开发的提案

首先，我很抱歉。我不想听起来冒犯，也无意针对任何人，因为正如伟大的智者 Veronica 所说，你的工具挺好的。

在很长一段时间里，我都坐在键盘后面敲代码。刚步入职场时，我决心遵循最佳实践，避免犯当时所用语言的 10 个经典错误，我就这样坚持了很长一段时间。

突然，我想涉足 Web 开发，但我很快就厌倦了继续敲击键盘、使用相同的理论并重复我在办公室里所做的事情。我想要一些新的、容易上手的东西。

现在，求者必得，我发现了许多神奇的工具，这些代码生成器让开发过程变得非常有趣，并产生了一种默契和火花。

于是我决定回到学术界，告诉新一代开发者有这些神奇工具的存在……我甚至多次受邀演讲……甚至走出了自己的城市。

紧接着，我加入了一个西班牙语社区，录制了一些视频并上传到 YouTube，并花了一些时间向新手解释——那些因为兴趣、好奇心，以及作为自己问题解决方案的寻找者而进入编程领域的人。

我为这个社区写了几篇文章，但我依然有着作为教师的渴望：我想告诉新一代开发者轮子是如何制造的、如何设计的，以及它们的内部结构是怎样的。

由于我们没有达成一致，我决定自己单干。我造了几个与那个曾带给我许多温暖的社区里类似的轮子，并试图复制一个，但效果并没有那么好。毕竟，他们领先了我好几年。

在半路上停下来休息时，有一天我买了一门关于现代 PHP 和 SOLID 原则的课程，然后读了一些关于架构、CQRS 模式和可维护代码的书籍。

于是，我决定精简自己，回归到一个任何人都能理解的简单想法。

经过不断的梳理，我突然灵光一闪。我得出了一个结论：我们所做的一切无非就是文本处理。这不算什么新鲜事吧？如果你有 Unix 背景，这很好理解。但是路上行走的普通人，那些接收着当红神奇工具广告的普通大众……能理解这一点吗？

这就是将整个框架塞进一张不大于餐巾纸的图表中的想法的由来。

但这绝不是一个玩具，因为 Secure Code Warrior 的课程深深打动了我。它必须是安全的、高质量的、可测试的，尤其是易懂且可扩展的。

对于一个小图表来说，这有很多限制，不是吗？

第一步，请求集中器：前端控制器 (front controller)。它有一个好听的名字，但无非是告诉服务器将所有到达的请求都指向同一个文件（一个简单的 index.php）。

好了，所有流量都进入集中器。

第二步，每个请求只有一个单一职责（SOLID 中的 S），在这里我简化到了极致：一个处理器 (handler)。仅此而已。

我稍作停顿，因为处理器 (handler) 接收请求 (request) 中的任何内容，进行处理并返回（文本），也就是说，它生成响应 (response)。

理论上……这就是全部了！

但是 Nelson！你明明说过会有安全性、可扩展性、代码质量以及可测试性！

SOLID 原则之一是关于使用接口 (Interfaces) 的，接口无非是一种看似无用的蓝图，但它包含契约（方法），使用它们的人必须遵守（实现）。

那么接口有什么用呢？简单来说，在我看来，它是创建具有相似行为但行为处理上有些许差异的物体的基础（多态性）。

接口允许什么？强制实现可扩展性和标准化！

现在，有一些基本的组件。我们需要负责定义用户可以访问的路由的东西（即路由器 Router）。我们需要负责根据路由列表分发到达的请求或触发错误消息的东西（即分发器），在我的情况下，我决定将其称为内核 (Kernel)。

如果您曾经使用过 .NET、Java 或 JSP，您会认为存在名为 Request（请求）和 Response（响应）的东西，它们分别对应“客户端的请求”和“我们返回的响应”。

请求的例子：
- 一个简单的链接
- 带参数的链接
- 包含文本的表单
- 带附件的表单

响应的例子：
- 状态码
- 纯文本
- HTML
- JSON
- XML
- 附件
- 重定向
- 等等，等等，等等。

那么安全性呢？
有一天我意识到，如果一个请求要进入，必须有一组条件来决定它是否能被处理。这就是宫殿的守卫，或者用工程语言来说，就是中间件 (middleware)（就像三明治里的汉堡肉），不穿过它，你就无法到达面包的另一侧。

中间件做什么？它检查请求，进行验证，修改或清理内容，帮助消除攻击向量。

但是请求中间可能会有一连串守卫试图阻止它，或者根本没有，这完全取决于请求的复杂程度以及它期望得到响应的目的地。

因此，每当我定义一条路由时，我都会说：看，如果有人带着去 A 门的请求，从黄色通道 (GET) 过来，而且那个门不需要任何守卫，那么我就不设任何障碍，并给他一张去该门的地图（负责接收并处理请求的处理器 handler）。

相反，如果有人带着去 B 门的请求，而这个门需要你持有印有“Wonka”字样的金色门票，那么我就设立一个看门人，询问并检查你的金色门票（一个中间件 middleware）。如果看门人决定让你通过，那么我就把去 B 门的地图交给他（负责处理并向你返回内容的处理器 handler）。

每个处理器 (handler) 都会接收请求及必要的细节，并进行相应处理，但它总是、总是、总是会发送一种派生自另一个接口的响应类型：Response。

现在响应是可扩展的；只要符合契约（Response 接口的方法），你就可以创建你需要的任何类型的响应。

为了避免不必要的 CPU 消耗，如果看门人决定你不符合条件，他就会一笔勾销，在转瞬之间将你排除，绝不回头。该请求及其历史记录就会消失在时间中。

那么，在这里我用餐巾纸上的解释来收尾，以下是按顺序进行的步骤：

1. 请求 (Request)
2. 前端控制器 (Front Controller)
3. 加载路由 (方法, 路由, 负责的处理器, 中间件列表) (Route)
4. 分发器 (Kernel)，它使用路由列表来匹配请求。如果存在中间件，则按顺序执行它们。如果所有中间件都允许，则继续；如果有一个不同意，则立即切断电路。
5. 输出 (Response)

这就是 Parina 框架，它是一个线性、严格、可扩展、高度可测试且对后续维护者极其友好的系统，因为每个请求只能由具有单一 handle 方法的处理器响应。

这就是它能写在一张餐巾纸上的原因。它在心智上非常容易消化，每当你觉得需要改进某些东西时，你的大脑肯定会得出同样的结论：让我们再添加一个中间件，因为它们负责流程控制、拦截危险输入、限制每秒过多请求，或者重定向未授权或无权限的用户。

现在，那张小纸片给我带来了意想不到的启示：一个极快的请求/响应循环，以及微乎其微的内存占用。

在 2003 年的一台配有 512MB 内存和赛扬 (Celeron) 处理器的机器上，运行 Parina 的服务器可以每秒响应超过 200 个请求，而消耗的内存仅为约 40KB。

因此我得出结论，我在概念上拥有了一个清晰、安全、可测试且易于理解的工具。而且，由于它所需的资源极少，它可以成为被行业遗忘的人们的工具，适用于互联网连接缓慢的社区，或者有着 15 年以上机龄电脑的人们。
